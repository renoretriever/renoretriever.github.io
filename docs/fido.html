FIDO (Fast IDentity Online)

基礎

  • FIDO 認証 3 つのポイント
    1. サーバに秘密、生体情報を持たなくて良い
      • パスワードで問題となったなりすましを原理的に防止
      • 秘密情報の管理にかかるコストを低減
    2. 必要な認証手段を後から容易に追加可能
      • 異なる手段の認証手段を追加してもパッケージの再導入は不要
      • 方式選定に迷う必要はなく、スピーディーな立上げが可能
    3. 充実した認定制度と多数の製品ラインナップ
      • FIDO 標準仕様への準拠が担保された製品を利用することにより、導入時の構築コストは最低限、品質も担保される

認証モデル

  • 利用者のデバイスなど手元にある 認証器 (Authenticator) が、利用者の本人性を検証する機能を持つ
    • 認証器による利用者の検証結果は認証サーバに送付され、認証サーバは検証結果の妥当性を確認し、認証が完結する
  • FIDO 認証では、ネットワーク上にクレデンシャル情報が流れることはない
    • 偽のサイトに誘導して利用者のパスワードを盗み出すようなフィッシング攻撃に耐性がある

登録の手順

  1. 認証器 => RP
    • ユーザが RP にアクセスし、登録のリクエストを送る
    • このときの認証には従来の ID/パスワードが使用
  2. RP => 認証サーバ
    • RP が、認証サーバにリクエストを送る
  3. 認証サーバ
    • 登録時に使用するチャレンジを生成
  4. 認証サーバ => 認証器
    • あらかじめ定めている認証ポリシーとチャレンジを送る
  5. ユーザ => 認証器
    • ポリシーをもとに、ユーザに認証を求める
    • 生体認証の場合は、ここで認証器が使用される
  6. 認証器
    • 認証で使用する公開鍵・秘密鍵のペアと、鍵 ID を生成
  7. 認証器 => 認証サーバ
    • 生成した認証用の公開鍵と鍵 ID などを、あらかじめアプリが用意しているアテステーション秘密鍵で署名し、署名データを認証サーバに送信
  8. 認証サーバ
    • 認証サーバがあらかじめ用意しているアテステーション公開鍵で署名を検証し、送信された公開鍵と鍵 ID を保存する
  9. RP => 認証サーバ
    • RP が結果を問い合わせ、登録を確認する

認証の手順

  1. 認証器 => RP
    • ユーザが、ログインのリクエストを送る
  2. RP => 認証サーバ
    • RP が、認証サーバにリクエストを送る
  3. 認証サーバ
    • 認証サーバが認証で使用するチャレンジを生成する
  4. 認証サーバ => 認証器
    • あらかじめ定めている認証ポリシーとチャレンジを送る
  5. ユーザ => 認証器
    • ポリシーをもとに、ユーザに認証を求める
    • 生体認証の場合は、ここで認証器による生体認証を行う
  6. 認証器 => 認証サーバ
    • チャレンジを認証用の秘密鍵で署名し、認証サーバに送付する
  7. 認証サーバ
    • 認証サーバがあらかじめ用意しているアテステーション公開鍵で署名を検証し、送信された公開鍵と鍵 ID を保存する
    • 認証サーバが認証用の公開鍵で署名を検証
  8. RP => 認証サーバ
    • RP が結果を問い合わせ認証を確認する

HTTP(HyperText Transfer Protocol)

  • 1989 年 3 月に誕生

  • CERN(欧州素粒子物理学研究所) のティム・バーナード・リーが考案

  • 特徴

    • ステートレス
      • ステートフルを実現するために、Cookie などが導入

  • HTTP/1.1 と一部の HTTP/1.0 では

  • 通信量の節約技術

    • 持続的接続(Persistent Connections)
      • 一定期間の間、TCP 接続を繋げ続ける
      • HTTP/1.1 から標準
    • パイプライン化(HTTP pipelining)
      • 従来はリクエスト受信後にレスポンスを受信するのを待って次のリクエストを発行していたものを、レスポンスを待たずに次のリクエストを発行できる

  • レスポンスの中身

    • ステータス
      • 結果の正常、エラーなどのステータスが入る
      • HTTP ステータスコード
    • メッセージヘッダー
      • ブラウザに詳細な情報を伝える
    • メッセージボディ
      • HTML など

メソッド

  • GET

    • リソースの取得

  • POST

    • エンティティボディ(メッセージボディ)の転送
    • GET でも転送可能だけど POST を使う

  • GET と POST の使い分け ※ RFC 2616 9 章および 15 章

    • GET メソッドは参照(リソースの取得)のみに用いる
    • GET メソッドは副作用が無いことが期待される
      • 副作用
        • サーバ側でのデータの追加・更新・削除が起こる作用
    • 秘密情報の送信には POST メソッドを用いること
      • URL 上に指定されたパラメータが Referer 経由で外部に漏洩する
      • URL 上に指定されたパラメータがアクセスログに残る
    • 送信するデータ総量が多い場合には、 POST メソッドを利用する
      • ブラウザやサーバが処理できる URL の長さには上限がある

  • PUT

    • ファイル転送
    • HTTP/1.1 の PUT 自体には認証機能がないため一般的な Web サイトでは使われない
    • REST を利用する時に使われる

  • HEAD

    • メッセージヘッダーの取得
    • URI の有効性やリソースの更新時間を確かめる目的で利用

  • DELETE

    • ファイルの削除
    • HTTP/1.1 の DELETE 自体には認証機能がないため一般的な Web サイトでは使われない
    • REST を利用する時に使われる

  • OPTIONS

    • サポートしているメソッドの問い合わせ
    • レスポンス -> Allow: GET,HEAD,POST,OPTIONS,TRACE

  • TRACE

    • 経路の調査
      • proxy - web などの経路
    • Max-Fowards というヘッダーフィールドに数値を含めサーバを通過するごとにその数値を減らしていきます
    • セキュリティ面から無効にするのがよろし

  • CONNECT

    • プロキシへのトンネリング要求

HTTP メッセージ

  • メッセージヘッダ
    • サーバやクライアントが処理すべきリクエストやレスポンスの内容が属性など
    • 構造
      • リクエストライン(リクエスト) / ステータスライン(レスポンス)
        • リクエストライン
          • メソッド
          • リクエスト URI
          • HTTP バージョン
        • ステータスライン
          • レスポンス結果
          • ステータスコードと説明
          • HTTP バージョン
      • リクエストヘッダーフィールド / レスポンスヘッダーフィールド
      • 一般ヘッダーフィールド
      • エンティティヘッダーフィールド
      • その他
  • CR+LF
    • 改行
  • メッセージボディ
    • 転送されるデータ
    • メッセージボディとエンティティボディ
      • メッセージ
        • HTTP 通信での基本単位で、オクテットシーケンスからなり、通信を介して転送
      • エンティティ
        • リクエストやレスポンスのペイロードとして転送される情報
        • エンティティフィールドとエンティティボディからなる

HTTP メッセージ

  • Client => Server

    • リクエストメッセージ

  • Server => Client

    • レスポンスメッセージ

  • 構造

    • メッセージヘッダー
      • ブラウザに詳細な情報を伝える
      • User-Agent
      • Cookie
    • 空行(CR+LF)
    • メッセージボディ
      • ブラウザに入力した内容が入る
      • 転送されるべきデータ

  • メッセージボディとエンティティボディの違い

    • メッセージ
      • HTTP 通信での基本単位で、オクテットシーケンスからなり、通信を介して転送される
    • エンティティ
      • リクエストやレスポンスのペイロード(付加物) として転送される情報で、エンティティヘッダーフィールドとエンティティボディからなる
    • 転送コーディングが施された場合にのみ、エンティティボディの内容が変化する
コンテンツネゴシエーション

  • 日本版のブラウザでアクセスすれば日本語で、英語版のブラウザでアクセスすれば英語版のようにそれぞれ違った web ページを表示する仕組み

  • コンテンツネゴシエーション

  • Apache

    • Apache ネゴシエーションアルゴリズム

  • コンテンツネゴシエーションに失敗した場合には 406 が返され、リクエストは失敗に終わる

  • HTTP ヘッダーによる指定の方が文書内での meta 指定よりも優先度が高い

    1. HTTP における Content-Type ヘッダの charset パラメータ
    2. HTML 文書内の META 宣言および http-equiv 属性で設定された、Content-Type ヘッダの charset パラメータ
    3. HTML 文書内の各要素の charset 属性

    • Content-Encoding

      • サーバーとクライアントで調整

HTTPS

  • SSL/TLS 技術を利用し、HTTP の通信内容を暗号化する

  • HTTPS では、HTTP は SSL/TLS と通信し、SSL/TLS が TCP と通信するようになる

    | アプリケーション(HTTP) |
|        SSL/TLS         |
|          TCP           |
|          IP            |

  • 共通鍵暗号 と 公開鍵暗号方式 の両方を使用するハイブリット暗号システム

    1. 共通鍵暗号で利用する公開鍵暗号を使って安全に交換する
    2. その後の通信は安全な共通鍵暗号を使う
  • 通信の仕組み
    1. C => S
      • Client Hello メッセージを送信することで SSL/TLS 通信が開始
      • クライアントがサポートする SSL/TLS のバージョンを指定し、暗号スイート(Cipher Suite) と呼ばれるリスト(使用する暗号化のアルゴリズムや鍵のサイズ) などが含まれる
    2. S => C
      • SSL/TLS 通信が可能な場合には、Server Hello メッセージで応答
      • サーバーがクライアントに送付する暗号スイートの内容は、クライアントから受けた暗号スイートの内容から選ばれたもの
    3. S => C
      • Cerfiticate メッセージを送信
      • メッセージには公開鍵証明書が含まれている
    4. S => C
      • Server Hello Done メッセージを送信することで最初の SSL/TLS ネゴシエーション部分が終わったことを通知
    5. C => S
      • Client Key Exchange メッセージで応答
      • メッセージには通信を暗号化するのに使う プレマスタシークレット と呼ばれる暗号化の種が含まれている
      • このメッセージは公開鍵証明書から取り出した公開鍵で暗号化される
    6. C => S
      • Change Cipher Spec メッセージを送信
      • このメッセージ以降の通信は暗号鍵を使って通信を行うことを示す
    7. C => S
      • Finished メッセージを送信
      • ネゴシエーションが成功したかどうかは、サーバーがこのメッセージを正しく復号できたかどうかで決まる
    8. S => C
      • サーバーからも同様に Change Cipher Spec メッセージを送信
    9. S => C
      • Finished メッセージを送信
    10. C => S
      • HTTP リクエストを送信
    11. S => C
      • HTTP レスポンスを送信
    12. C => S
      • close_notify メッセージを送信して接続が close

Other

  • コンテンツコーディング

    • エンティティに適用するエンコーディングのことを指していて、エンティティの情報を保ったまま圧縮
    • 容量を小さくするための技法
      • gzip
      • compress
      • deflate(zlib)
      • identity(エンコーディング無し)

  • チャンク転送コーディング

    • リクエストしたリソースのすべてのエンティティボディの転送が完了しないとブラウザなどでは表示されない。
    • データをいくつかに分割する技術
    • ストリーミングダウンロード/アップロードと呼

  • MIME(Multipurpose Internet Mail Extensions)

    • メールでテキストや画像、動画と言った複数の異なるデータを扱うための仕組み
    • 画像などのバイナリデータを ASCII 文字列にエンコードする方法や、データの種類を表す方法などを規定
    • multipart/form-data
      • Web フォームからのファイルアップロードに使用される
    • multipart/byteranges
      • ステータスコード 206(Partial Content) レスポンスメッセージが複数の範囲の内容を含む時に使用される

  • レンジリクエスト

    • レジューム機能を利用
      • 以前ダウンロードを中断した箇所から、ダウンロードを再開できるもの
    • レンジリクエストを指定し、範囲指定を行う
    • 5,000 〜 10,000 バイト
      • Range: bytes=5000-10000
    • 5,000 以降
      • Range: bytes=5000-
    • 最初から 3,000 バイトまでと、5,000 〜 7,000 までの複数の範囲
      • Range: bytes=-3000,5000-7000
    • レスポンスは 206 Partial Contetn が返される
    • サーバ側がレンジリクエストに対応してない場合には、ステータスコード 200 OK というレスポンスメッセージで、完全なエンティティが返される
  • 認証方法(HTTP/1.1)
    • BASIC 認証
    • ダイジェスト認証
    • SSL クライアント認証
    • フォームベース認証

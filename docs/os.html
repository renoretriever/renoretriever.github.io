  • ロードアベレージ

    • CPU リソースに対する需要を示す指標で、実行中のスレッドと実行されるのを待ってキューイングされているスレッドの合計数から計算
    • ロードアベレージには、キューの長さをサンプリングせずに、使用率にスレッドスケジューラのレイテンシの合計を加える新しい計算方法もあり、その方が従来よりも精度が高くなっている
    • ロードアベレージの方が CPU の数よりも高ければ、スレッドを処理する CPU が足りておらず、一部のスレッドが待っていると考えればよい
    • ロードアベレージが CPU 数よりも小さければ、(おそらく)CPU に余裕があり、スレッドはほしいときに CPU を手に入れて実行されているのだろう
    • 3 つのロードアベレージの数字は、指数移動平均でり、1、5、15 分という時間を超える負荷を反映している
    • 新しいシステムの例で考えると、64 個の CPU を搭載したシステムでロードアベレージが 128 だったとする
      • これは、平均して各 CPU が 1 個のスレッドをいつも実行しており、1 個のスレッ ドがいつも待っているという意味になる
      • 同じシステムでロードアベレージが 10 なら、54 個の CPU バウンドのスレッドをさらに実行しなければすべての CPU がビジー状態にならないということであり、かなりの余裕がある
    • Linux のロードアベレージ
      • 現在のところ、Linux はロードアベレージに割り込み不能状態でディスク I/O を実行しているタスクを加えている
        • この値だけからは、CPU とディスクの負荷がどれくらい反映されているかはわからないので、ロードアベレージは CPU だけの余裕または飽和を意味するものとしては解釈できない
        • 時間の経過とともに CPU とディスクの負荷の割合がどのように変わるかがわからないので、3 つのロードアベテージの数値を比較するのも難しくなる
      • ほかのリソースの負荷も考えたいのなら、リソースタイプごとに別々のロードアベレージを計算するという方法もある
      • Linux で CPU の負荷を知りたい場合は、vmstat(1) や mpstat(1) が返す数字などのほかの指標を使った方がよい

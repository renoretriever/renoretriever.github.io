仮想化

仮想化技術

  • 準仮想化

    • 仮想マシンの性能を向上させるための技術
    • 物理マシンで動作する OS と同じ OS を仮想マシンで実行する完全仮想化を実現する技術が HVM(HardwareVirtualMachine) ですが、完全仮想化にはストレージやネットワークの I/O などの性能が劣化する問題がある
    • これらの I/O のみを仮想化に最適化するための手段が準仮想化で
    • 準仮想化の仮想マシン上で準仮想化に対応したカーネルを利用するほか、完全仮想化の仮想マシンに準仮想化対応デバイスドライバをインストールすることで性能を向上させる方法などがある
    • 完全仮想化＋準仮想化対応デバイスドライバの組み合わせで動作させることが多く、Linux カーネルにもいくつかの準仮想化ドライバがあらかじめ組み込まれています

  • コンテナ型仮想化

    • カーネル上で動作しているプロセスをコンテナという単位で分離することで、別々の環境として利用できるようにする技術
    • Linux では LXC などが利用
    • カーネルは単一のものを利用するため、メモリ使用量が少なく、性能劣化もほとんど無いのがメリット

仮想化ソリューション

  • ハイパーバイザー型

    • Xen

    • 通常の OS の代わりにハードウェア上で動作し、アプリケーションの代わりに仮想マシンを実行

    • ハイパーバイザーの環境構造

      +--------------------+
|  アプリケーション  |
+--------------------+
|      ゲストOS      |←※図の都合でOSが1つですが、実際には複数存在可能
+====================+
|  ハイパーバイザー  |
+--------------------+
|    ハードウェア    |
+--------------------+

  • コンテナ型

    • OpenVZ

  • ホスト OS 型

    • VirtualBox

  • ハイパーバイザ型/ホスト OS 型のハイブリッド

    • KVM
      • ハイパーバイザー型に近いが、仮想マシンが QEMU を利用していることや、仮想マシンがプロセスとして動作して見えることなどから、ハイパーバイザー型とホスト OS 型の中間ぐらいのハイブリット型とでも言うべき構造

HVM は完全仮想化と呼ばれる仮想マシンを実行し、この場合ゲストOSは通常と同じものが使用できます。PVは準仮想化と呼ばれ、OSにパッチを当てることで仮 想化環境に最適化を行いますが、オープンソースのOSか、ドライバのみ準仮想化するなどの方法をとる必要があります。

  • Linux で利用できるその他の仮想化ソリューション
    • OpenVZ
      • 「コンテナ型」と呼ばれる方式の仮想化技術を実装
      • コンテナ型の特徴として、各コンテナは独立した別々のシステムとして動作しますが、Linux カーネルは単独のものを共有
      • このような構造のメリットとして、仮想マシン型に比べると処理の切り替えのオーバーヘッドが少ないことが挙げられます
      • 一方、カーネルが単一であるということで、異なるカーネルを使い分けたり、Windows などの Linux 以外の OS を同時並行で動作させることができません
      • Web サーバーのホスティングサービスを行う場合などには、コンテナ型の仮想化ソリューションが適しています
      • OpenVZ
    • VirtualBox
      • Linux や Windows などのホスト OS 上でアプリケーションとして動作する方式の仮想化ソリューション
      • ホスト OS を必要とするところから「ホスト OS 型」と呼ばれます
      • ハイパーバイザー型同様、仮想マシンを実行する方式のため、異なる Linux カーネルや Windows などのOSを実行することができます
      • 日常的に使い慣れたホスト OS を利用できるため、使用に対するハードルが低いのも特徴
      • 一方で汎用的な OS 上でさらに OS を動かすことになるため、仮想ディスクや仮想ネットワークなどの I/O 周りの性能はハイパーバイザー型やコンテナ型に劣ることになる
      • それほど性能を必要としない場合や、開発・テスト環境の構築には、ホストOS型の仮想化ソリューションが適しているでしょう
      • VirtualBox
  • クラウド管理ツール
    • OpenStack, CloudStack
      • 独自のIaaS を構築、運用するためのクラウド管理ツール
      • 対応するハイパーバイザーなどは多岐に渡っており、企業のプライベートクラウド構築や商用クラウドサービスの提供などにも利用
    • Eucalyptus
      • AWS 互換の API を提供することを目的としたクラウド管理ツール
    • OpenNebula
      • Xen や KVM といったハイパーバイザーに対応したクラウド管理ツール
      • その他のクラウド管理ツールに比べてシンプルな構成であるのが特徴

Immutable Infrastructure

  • 従来のシステムのライフサイクル

    • 構成するハードウェアやソフトウェアの保守期限が重要な要素
    • システム構築を計画する際に企業のシステム部は、そのシステムが実現する業務プロセスやサービスに見合ったシステム投資額を決定し、そのシステムを何年利用するかを決める
    • アーキテクチャやソフトウェアはハードウェアの保守期限より早いライフサイクル

  • ビジネスに沿ったライフサイクル

    • ビジネススピードに沿って、素早く投入、素早く撤退(資産を残さない)を実現できるようにしたい
    • インフラ環境の構成情報をコードとして定義し、自動構築の仕組みを使って、構成を再現できるようにすることが必要
      • Infrastructure as Code
        • Chef, Puppet, Ansible, Serverspec

  • デプロイメント

    • インプレース更新
      • すでに動作している実行環境に、新しいアプリケーションモジュールを配布する方法
      • リクエストの受付が一時的にできなくなるという欠点がある
      • リリースするまでアプリケーションの動作は確認できないので、性能問題を引き起こすアプリケーションの問題を抱えていても、それが分かるのはリクエストが来てから
    • Blue-Green
      • アプリケーションの更新や、実行基盤の更新の際に、新しいバージョンのアプリケーションを実行環境毎に別に構築し、動作確認まで行った上で、現行の環境と URL スワップを行い、リクエストを受け付けられる状態を継続していく方法
      • 2 つの本番環境に相当する 2 つの環境を切り替えて利用

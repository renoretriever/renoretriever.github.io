サーバー

  • サーバラックに格納

    • 大抵のラック幅が 19 インチ
    • 高さは 4.5 cm 単位に 40 〜 46 目盛りついている
    • 1 目盛り = 1 U

  • サーバ設置時に確認すること

    • サーバのサイズ(U)
    • 消費電力(A)
    • 重量(kg)

  • 内部構造

    • HDD, CPU, メモリー, PCI, NIC, 電源 etc...
    • コンポーネント同士をつなぐ線を バス と呼ぶ
    • PCI Express スロット
      • 外部機器を接続する場所

  • I/O 制御(IA サーバアーキテクチャ)

    • I/O ハンドラ(IOH)
      • CPU に近い場所に置かれるため ノースブリッジ
    • I/O コントローラーハブ(ICH)
      • サウスブリッジ
    • これらを総称して チップセット と呼ぶ

  • 直列 / 並列

    • メリット
      • 直列
        • 仕組みがシンプルで設計や実装の難易度が低い
      • 並列
        • 複数のリソースを有効に活用でき、直列に比べて同じ時間で処理できる仕事量が増える
          • 1 処理の時間が短くなるわけではない
        • 1 つが故障しても処理を継続できる
    • デメリット
      • 直列
        • 複数のリソースを有効活用できない
      • 並列
        • 処理の分岐や合流のためのオーバーヘッドがある
        • 排他制御などを考慮する必要があり、仕組みが複雑になるため、設計や実装の難易度が高い

  • 同期 / 非同期

    • 同期
      • 処理を他に依頼した後、その処理が終わるまで待機を行っている状態
    • 非同期(asynchronous)
      • 処理を他に依頼した後、他の作業を継続して実施できる状態
      • 先に依頼した処理が終わったかどうかの確認が必要になる
      • デメリット
        • 障害時に処理が失われることがあること
        • 障害時にデータの整合性が失われることがあること
    • 非同期 + 順序保証(ordered)'
      • 災害対策サイトとのデータ同期や、マシン間での頻繁なデータやり取りで見られる

構築時の確認ポイント

  • テストサーバーの有無

  • メモリテスト

  • インストールパッケージ

    • ミドルウェア

  • ディスク

    • 容量
    • パーティション分割
      • /boot 配下は分ける
      • 負荷の観点からデータの更新が多い部分は分ける

  • ネットワーク

    • ホスト名
    • IP アドレスとネットワーク
    • ブロードキャストアドレス
    • デフォルトゲートウェイの IP アドレス
    • ドメインネームサーバの IP アドレス
    • 本番のネットワーク構成とテスト環境のネットワーク構成
    • ルーティング

  • C10K 問題

    • ハードウェアの性能上は問題がなくても、あまりにもクライアントの数が多くなるとサーバがパンクする問題のこと
    • OS のファイルディスクリプタやプロセス数の上限に達してしまったり、1 プロセスあたりの消費するメモリサイズはたいしたことがないですが、万レベルになると、コンテキストスイッチなどの負荷もばかないならない。
    • ノンブロッキング I/O
      • node.js, nginx, ClassFish
  • ファイルディスクリプタ
    • プログラムがアクセスするファイルや標準入出力などを OS が識別するために用いる識別子
    • 上限値がある
      • 1 プロセスで開ける上限値
        • ソフトリミット
        • ハードリミット
      • システム全体で開ける上限値
        • fs.file-max
    • ulimit コマンド

  • キュー

  • 排他制御

    • 複数人でリソースを共有している場合、誰かがそのリソースを利用しているときには、他の人がそのリソースを利用できないようにする仕組み
    • 直列処理では不要だが、並列処理では必要
    • 複数の処理の流れから共有資源(CPU, メモリ, ディスク etc...) に同時アクセス(主に更新) すると不整合が発生するため排他制御で保護する必要がある
    • DB
      • スピンロック
        • CPU でスピン(ループ) して特に意味のない処理をして待ち、ロック解放を待つ
        • スリープして待つとコンテキストスイッチが高くなるのでスピンロックが良い
    • OS
      • 昔
        • ビッグカーネルロック(BKL) と呼ばれる 1 つのスピンロックで守られていた
  • ステートフル/ステートレス
    • ステートフル
      • 情報を維持し、次に引き継ぐことができる
      • いまの状態を考慮
    • ステートレス
      • いまの状態を考慮しない

  • 可変長 / 固定長

    • 可変長
      • スペースを有効活用できるが性能面で不安定な傾向がある
    • 固定長
      • 無駄なスペースができうｒが性能面で安定している傾向がある

  • データ構造(配列と連携リスト)

    • データの整理方法
    • 配列
      • 同じ形の箱を隙間なく順番に並べたようなデータ構造
      • 番号が分かればデータにすぐにアクセス可能
      • データの挿入・削除が遅いデータ構造
        • 挿入するときには後のデータをずらす必要がある
        • 削除の場合には後のデータをつめる必要がある
    • 連結リスト
      • 箱が紐でつながれたようなデータ構造
      • 次の箱の位置情報を持っている
      • データの挿入・削除が早いデータ構造
    • ハッシュテーブル
      • キーと値がペアの表になっているデータ構造
      • キーはハッシュ関数でハッシュ値に変換される
      • データ量が増えても基本的にイコール検索の速さは変わらない
      • 範囲検索は不得手
  • 探索アルゴリズム(ハッシュ/ツリーなど)
    • データを探す方法
    • B-Tree インデックス
  • ポーリング
    • 定期的な問い合わせ
    • 特徴
      • 問いかけが一方向
      • 問いかけは定期的であり、一定間隔で行われる
    • 例
      • NTP
  • 耐障害性 / 高可用性
    • 耐障害性
      • アーキテクチャ内のいずれかのコンポーネントで障害が起きても、システムはパフォーマンスを低下させることなく機能し続ける
      • 障害がおきても SLA を守る
    • 高可用性
      • 停止時間をいかに短くするか
      • アーキテクチャ内のいずれかのコンポーネントで障害が起きても、システムは機能し続ける
  • 冗長化
  • 障害対応

移設/移行作業

--

注意点

  • 切り戻し手順は必ずつくる
  • 作業時間は必ずつくる
  • 誰が担当するのかははっきりさせておく
  • いろいろ時間がかかるので、計画的にやる

事前作業

  • コンテンツ等は先に同期しておく
    • 所有者、権限、シンボリックリンクなどの確認を行う
    • rsync
  • 作業手順の作成
    • 作業内容
    • 作業詳細
    • 所要時間
    • 作業開始時間
    • 担当者
  • 切り戻し判断基準の作成
  • TTL を事前に短くしておく
  • 外向きの IP が変わる可能性があるので、外部連携しているシステムへの事前連絡
  • バッチ処理などは停止 or 時間をずらすなどの操作
    • cron 止めて良いかも

作業開始

  • メンテナンス画面への切り替え

    • 社内は開放している可能性があるので、別回線やスマホから確認する

  • 最終同期

    • コンテンツ
    • 設定ファイル
    • DB なら dump ファイル

  • 新サーバ側の DB 接続先情報のなど変更

  • hosts 切り替えなどで動作確認

    • アクセスログからリクエストが来ていることを確認

  • hosts ファイルから削除

  • DNS 切り替え

    • 新環境でアクセスがきていることを確認
    • 旧環境ではアクセスがきてないことを確認

作業終了

  • 1 週間程度は平行運用して、その後旧環境を close
